Step 02

예측 목표 및 변수 설계

예측 목표
특정 예매 건이 봇·매크로에 의한 비정상 예매인지 여부 → 이진 분류 (0: 정상 / 1: 봇)
G1
예매 행동 패턴 — 플랫폼별 UX가 달라 수집 변수가 다름
인터파크팝업창 진입 속도 · 좌석 클릭 패턴 · 좌석 변경 횟수
YES24대기열 진입 타이밍 · 등급 선택 체류 시간 · 등급 변경 횟수
멜론예매버튼 반응 시간 · 보안문자 입력 속도 · 5분 내 결제 완료율
G2
계정 이력 — 3개 플랫폼 공통
최근 30일 예매 취소율 · 동일 공연 중복 시도 횟수 · 계정 생성일 · 결제 수단 다양성
G3
공연 맥락·인기도 — 3개 플랫폼 공통
아티스트 SNS 언급량 · 티켓 예상 매진 시간 · 티켓 가격 · 공연 장르
Step 03

노드별 데이터셋 구성 (DCAT)

노드 1 — 인터파크
K-pop 콘서트·스포츠 예매 행동 데이터셋

Non-IID: 팬덤 집중 트래픽, 밀리초 단위 매크로 지배적
노드 2 — YES24
뮤지컬·클래식 예매 및 계정 이력 종합 데이터셋

Non-IID: 고가 티켓 위주, 대기열 기반 봇 패턴 특화
노드 3 — 멜론
음원 스트리밍 연계 공연 예매 데이터셋

Non-IID: 버튼 반응 시간 분포가 타 플랫폼과 완전히 다름
Non-IID 종합: 장르·팬층이 달라 봇 패턴 분포가 다름. 연합학습으로 합쳐야 범용 탐지 모델 완성.
Step 04

학습 시나리오 및 글로벌 모델

핵심 원리
데이터는 각 플랫폼 서버에 그대로 두고, 학습된 가중치(패턴 요약본)만 중앙 서버로 전송 → FedAvg 알고리즘으로 병합 → 글로벌 모델 완성
1
서버 선정
DCAT로 3개 플랫폼 노드 선정
2
모델 배포
초기 모델을 각 플랫폼에 배포
3
로컬 학습
자사 데이터로 학습, 원본 미전송
4
가중치 업로드
가중치만 중앙 서버 전송
5
FedAvg 집계
병합 → 글로벌 모델 완성
Step 05

기대 효과 및 차별점

①탐지 정확도 향상
K-pop 매크로·뮤지컬 봇·CAPTCHA 우회 봇 전 유형 학습. 플랫폼 간 봇 패턴 공유 탐지 가능.
②실사용자 보호
오픈 순간부터 이상 행동을 실시간 감지해 봇 차단. 수 초 만에 전석 매진 피해 방지.
③지속 학습·업데이트
새 봇 패턴 탐지 시 로컬 재학습 후 가중치 전송으로 글로벌 모델 빠르게 갱신.
④데이터 주권 보호
원본 데이터 외부 미전송. 영업기밀 노출 없이 공동 대응 가능한 유일한 구조.
단독 플랫폼중앙 집중연합학습 (본 시나리오)
봇 탐지 범위자사만전체 가능✓ 전체 가능
데이터 공유불필요원본 필요✓ 가중치만
개인정보·영업기밀보호✗ 위험✓ 완전 보호
스키마 충돌없음✗ 발생✓ 우회 가능
결론: 연합학습만이 데이터 주권 보호 + 전 플랫폼 공동 대응 + 지속 업데이트를 동시에 실현하는 유일한 방법이다.
