Step 01

참여 주체와 한계점 정의

누가 데이터를 가지고 있고, 왜 한곳에 모으는 것이 현실적으로 불가능한지 설명합니다.

Node 1
인터파크티켓
K-pop 콘서트·스포츠 이벤트 위주. 팝업창 기반 좌석 직접 선택 방식.
Node 2
YES24 공연
뮤지컬·클래식·연극 고가 공연 위주. 대기열 번호 발급 후 등급 선택 방식.
Node 3
멜론티켓
음원 스트리밍 연계 공연 예매. 카운트다운 자동전환·보안문자(CAPTCHA) 방식.
🔐
한계 1
직접 경쟁 관계 → 영업기밀
세 플랫폼은 서로 직접 경쟁 관계다. 예매 행동 데이터(어떤 팬이 몇 초에 예매하는지, 취소율 등)는 핵심 영업기밀이어서 타사에 제공하는 것이 구조적으로 불가능하다.
⚖️
한계 2
개인정보보호법
회원의 예매 이력·클릭 패턴·계정 정보는 개인정보보호법상 민감 개인정보다. 제3자 서버로 이전하려면 별도 동의와 법적 절차가 필요해 현실적으로 불가능하다.
🔧
한계 3
스키마 구조 차이 → 연합학습이 해답
각 플랫폼의 데이터 컬럼 구조가 달라 원본 통합 시 구조 충돌 발생. 연합학습은 가중치만 공유하므로 스키마가 달라도 우회 가능하다.

YES24

(captcha 없음)

세 가지 한계점은 각각 영업적·법적·기술적으로 "데이터는 제자리에 두고, 가중치만 공유"하는 연합학습의 필요성을 뒷받침한다.
Step 02

예측 목표와 변수 설계

AI 모델이 맞추고자 하는 '결과'와 이를 위해 필요한 '힌트'들을 정의합니다.

예측 목표
특정 예매 건이 암표 재판매 목적 봇·매크로에 의한 비정상 예매인지 여부 → 이진 분류 (0: 정상 예매 / 1: 봇 예매)
G1
예매 행동 패턴
플랫폼별 예매 UX가 달라 수집 변수가 다름 — 봇 이상 징후가 가장 뚜렷하게 나타남
인터파크티켓
팝업창 기반 좌석 직접 클릭
공연 선택›팝업 진입›좌석 클릭›결제
G1-A
팝업창 진입 속도
오픈~팝업까지 ms. 봇은 수백ms 이내 자동 진입.
G1-B
좌석 클릭 패턴
도면 내 클릭 경로·속도. 봇은 특정 구역 반복 클릭.
G1-C
좌석 변경 횟수
재선택 횟수. 봇은 스캔하듯 반복 변경.
YES24 공연
대기열 번호 발급 → 등급 선택
공연 선택›대기열 발급›등급 선택›결제
G1-A
대기열 진입 타이밍
오픈 대비 진입 시간. 봇은 오픈 직전 자동 진입.
G1-B
등급 선택 체류 시간
VIP·R·S 화면 체류. 봇은 0ms대 즉시 선택.
G1-C
등급 변경 횟수
재선택 수. 봇은 즉각 차선 등급 전환.
멜론티켓
카운트다운 자동전환·보안문자
공연 선택›카운트다운›CAPTCHA›결제
G1-A
예매버튼 반응 시간
활성화~클릭 ms. 봇은 0~10ms 자동 클릭.
G1-B
보안문자 입력 속도
CAPTCHA 완료. 사람 3~10초, 봇은 즉각 처리.
G1-C
5분 내 결제 완료율
선점 후 결제. 봇은 자동 결제로 거의 100%.
G2
계정 이력
3개 플랫폼 공통 — 봇 계정은 신규 생성·단기 사용 패턴이 뚜렷
📊
최근 30일 예매 취소율
봇은 결제 후 취소 반복 또는 대량 취소 패턴
🔄
동일 공연 중복 시도 횟수
같은 공연 반복 접근. 봇은 실패 시 즉각 재시도
📅
계정 생성일
봇은 예매 오픈 직전 생성된 신규 계정 다수 사용
💳
결제 수단 다양성
봇은 자동화된 단일 결제 수단만 반복 사용
G3
공연 맥락·인기도
3개 플랫폼 공통 — 봇은 인기 공연·고가 티켓에 집중
📱
아티스트 SNS 언급량
버즈량 높을수록 봇 유입 가능성 높음
⏱️
티켓 예상 매진 시간
매진 빠를수록 차익 크므로 봇 집중도 높음
💰
티켓 가격
고가 티켓일수록 봇 유인과 강한 양의 상관관계
🎭
공연 장르
K-pop·뮤지컬 등 장르별로 봇 패턴이 다름
Step 03

참여 노드별 데이터셋(DCAT) 구성

3개 플랫폼 노드별 데이터셋 구성을 요약하고, Non-IID 특성을 설명합니다.

노드 1 — 인터파크티켓
'K-pop 콘서트·스포츠 예매 행동 데이터셋'
G1G2G3
G1: 팝업창 진입 속도 · 좌석 클릭 패턴 · 좌석 변경 횟수
봇 패턴 다양성
K-pop 팬덤 집중 트래픽. 밀리초 단위 매크로 패턴 지배적. 스포츠 시즌 편향 존재.
샘플: 오픈 직후 수초 내 예매 집중 → 극단적 행동 패턴 분포
노드 2 — YES24 공연
'뮤지컬·클래식 예매 및 계정 이력 종합 데이터셋'
G1G2G3
G1: 대기열 진입 타이밍 · 등급 선택 체류 · 등급 변경 횟수
봇 패턴 다양성
고가 뮤지컬 티켓 위주. 암표 피해 금액 최대. 대기열 기반 봇 패턴 특화.
샘플: 장기 회원 비중 높아 정상·봇 레이블 구분 명확
노드 3 — 멜론티켓
'음원 스트리밍 연계 공연 예매 데이터셋'
G1G2G3
G1: 예매버튼 반응 시간 · 보안문자 입력 속도 · 5분 결제 완료율
봇 패턴 다양성
카운트다운 방식 → 버튼 반응 시간 분포가 타 플랫폼과 완전히 다름.
샘플: CAPTCHA 우회 봇 패턴 특화. 스트리밍 인기지수 기반 봇 집중.
Non-IID 종합: 세 플랫폼 장르·팬층이 달라 봇 패턴 분포가 다름. G1 수집 맥락 자체가 달라 데이터 불균형 두드러짐. 연합학습으로 합쳐야 범용 탐지 모델 완성.

Step 05

기대 효과 및 차별점

이 시나리오로 무엇을 목적으로 학습시켰고, 그 결과 어떤 효과가 나타나는지를 설명합니다.

목
학습 목적
암표 봇 이진 분류 모델 학습
세 플랫폼이 G1·G2·G3 데이터로 예매 건이 정상인지 봇인지 판별하는 모델을 연합학습 방식으로 훈련. 원본 데이터 공유 없이 자체 학습만 수행.
합

결
4가지 기대 결과
🎯
①탐지 정확도 향상
단독 모델은 자사 봇만 탐지. 글로벌 모델은 K-pop 매크로·뮤지컬 봇·CAPTCHA 우회 봇 전 유형 학습. 인터파크에서 탐지된 패턴이 YES24·멜론에서도 차단.
🛡️
②실사용자 보호
예매 오픈 순간부터 이상 행동 패턴을 실시간 감지해 봇 차단. 수 초 만에 전석 매진되는 피해를 방지하고 실제 팬이 정상 예매할 수 있는 환경 조성.
🔄
③지속 학습·빠른 업데이트
새로운 봇 패턴 탐지 시 로컬 재학습 후 가중치 전송으로 글로벌 모델 빠르게 갱신. 단독 대응보다 훨씬 빠른 업데이트 사이클 가능.
🔒
④데이터 주권 보호
원본 예매 데이터는 각 서버 밖으로 나가지 않아 개인정보 침해 위험 없음. 영업기밀 노출 없이 공동 대응 가능한 유일한 구조.

차별점 — 왜 연합학습이어야 하는가?

단독 플랫폼 모델중앙 집중식 학습연합학습 (본 시나리오)
봇 탐지 범위자사 봇만 탐지전 플랫폼 가능✓ 전 플랫폼 가능
데이터 공유불필요원본 공유 필요✓ 가중치만 공유
개인정보 보호가능✗ 침해 위험✓ 완전 보호
영업기밀 보호가능✗ 노출 위험✓ 완전 보호
스키마 충돌없음✗ 발생✓ 우회 가능
봇 수법 대응느린 단독 대응가능하나 법적 제약✓ 빠른 공동 갱신
결론: 단독 모델은 자사 봇만 알고, 중앙 집중은 영업적·법적·기술적으로 3중 불가. 연합학습만이 데이터 주권 보호 + 전 플랫폼 공동 대응 + 지속 업데이트를 동시에 실현하는 유일한 방법이다.
